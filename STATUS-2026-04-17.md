## What Is This Project?

**The Pour List** — a crowd-sourced happy hour directory. Users scan happy hour menus with their phone camera, AI extracts the text, and the parsed menu is stored permanently (photo discarded). No accounts. Anonymous device hash for spam prevention. Designed to scale nationally.

- **GitHub:** github.com/xalanx2000x/pourlist
- **Dev:** localhost:3000 (`npm run dev`)
- **Prod:** https://pourlist.app

## Database Schema (as of 2026-04-17, plus additions)

### venues
```sql
id, name, address_backup, lat, lng, zip, phone, website, type,
status: 'unverified'|'verified'|'stale'|'closed',
contributor_trust: 'new'|'trusted',
last_verified, last_flag_decay_at,
menu_text, menu_text_updated_at, latest_menu_image_url,
photo_count, created_at,
-- added:
hh_summary, hh_type, hh_days, hh_start, hh_end,
hh_type_2, hh_days_2, hh_start_2, hh_end_2,
hh_type_3, hh_days_3, hh_start_3, hh_end_3
```

### venue_events (added 2026-04-23)
```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
venue_id UUID REFERENCES venues(id) ON DELETE CASCADE NOT NULL,
event_type TEXT NOT NULL,  -- 'view', 'hh_confirm', 'photo_upload'
device_hash TEXT,
lat NUMERIC, lng NUMERIC,
created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
```
Indexes: `(venue_id, created_at DESC)`, `event_type`, `device_hash`

### flags
```sql
id, venue_id, photo_id (legacy), reason, device_hash,
active BOOLEAN NOT NULL DEFAULT TRUE,
lat DOUBLE PRECISION, lng DOUBLE PRECISION, created_at
```

### venue_flag_events
```sql
id, venue_id, device_hash, action ('flag'|'confirm'|'reopen'),
UNIQUE(venue_id, device_hash, action)
```

### device_stats
```sql
device_hash PK, submission_count INT, updated_at
```

### photo_sets
```sql
id, venue_id, photo_urls (text[]), uploader_device_hash, created_at
4-photo limit: oldest purged on new insert
```

## Moderation Rules (FINALIZED)

| Rule | Value |
|------|-------|
| Flag GPS required | YES — 10m Haversine |
| Confirm GPS required | YES — 10m Haversine |
| N=2 distinct devices | status: `'stale'` |
| N=4 distinct devices | status: `'closed'` |
| Trusted flag weight | 2x toward N threshold |
| Trusted threshold | 10 submissions → 'trusted' |
| Flag decay | 1/month per venue |
| Menu commit | Clears all flags |

## What Changed Since 2026-04-17

### iOS Safe Area (59d6ccc)
```css
body { padding-top: env(safe-area-inset-top); padding-bottom: env(safe-area-inset-bottom); }
```
Pushes UI below Dynamic Island and above home indicator.

### Swipe-to-close fix (3dc78ff)
VenueDetail `handleTouchMove` was calling `querySelector('.overflow-y-auto')` on the wrong element → swipe never tracked. Fixed: use `e.currentTarget` directly.

Also added `overscroll-behavior: none` to html+body to kill iOS rubber-band pull-to-refresh that was competing with swipe-to-close.

### Map/List Symmetry (bd0d494)
Both map and list now receive the same `visibleVenues` filtered by `currentBounds = listBounds ?? mapBounds`. Map and list are always in sync — panning map filters list, switching views preserves the last active map area.

### Dynamic Area Name (bd0d494 + 96836f9)
Reverse geocoding via Mapbox API on first user location resolve → `areaName` displayed in list header. When user has panned the map, shows "swipe map for full list" hint.

### venue_events Tracking (fc02a0b)
New `trackVenueEvent` helper (`src/lib/track-venue-event.ts`). Events wired:
- `view`: when user taps a venue
- `photo_upload`: when menu photos are submitted
- `hh_confirm`: when HH schedule is included in submission

Migration run by Tyler in Supabase SQL Editor 2026-04-23.

### Parser: "midnight" + "til" support (682d3dc)
`parseTimeToMin('midnight')` → 0 (midnight = 12:00 AM). "til" normalized digit-aware: "4 til 7" → range separator. "10pm til midnight" → type=late_night, end=null (close).

### HH window key fix (86a96d0 + earlier fix)
`appendWindow` was hardcoding `hh_start` and `hh_end` for ALL windows — window 2's `startMin=0` overwrote window 1's `startMin=840`. Fixed to use `prefix.replace('type', 'start')` and `replace('type', 'end')` so window 2 gets `hh_start_2`.

### MenuCapture back button (5717358)
Changed "← Back" text to "←" icon in a circle — text was overflowing the small button. Also combined camera + gallery into single "📷 Take or Choose Photo" button (removed separate Files upload).

## Critical Issue: New Venue Creation Flow

**Problem identified 2026-04-23 — not yet fixed, pending Tyler's design decision**

Current two-step flow: `create-venue` → `commit-menu`
- `create-venue`: no name dedup, creates bare venue (no HH data)
- `commit-menu`: has geo-dedup code but it's DEAD CODE because page.tsx always passes a venueId (bypassing the dedup path)
- Photo upload failure silently swallowed (user never told)
- No rollback if commit-menu fails after create-venue succeeds → orphan venue
- `menuText` from AI photo parsing never written to `menu_text` column

**Proposed redesign:** Single `/api/submit-venue` endpoint:
- One call instead of two
- Name dedup + geo-dedup in one place
- Rollback venue if photo upload fails
- Returns photo success/failure to client explicitly
- Stable coordinates: user-submitted address → geocoded (not GPS drift)

Tyler is thinking about the design. Key open question: venue coordinates from address (more accurate, more friction) or from EXIF GPS (automatic, can drift)?
